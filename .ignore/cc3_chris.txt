  To Run on Windows

  # Clone and build
  git clone https://github.com/chrishayuk/larql.git
  cd larql
  cargo build --release

  # The binary will be at:
  # target\release\larql.exe

  What Works

  - All browse operations: DESCRIBE, WALK, SELECT, STATS
  - CPU inference: INFER works but without Metal GPU acceleration
  - Vindex extraction: From HuggingFace or GGUF
  - LQL REPL: Full query language support
  - HTTP server: larql serve

  What's Different

  1. No Metal GPU — Windows uses CPU-only BLAS. Inference will be slower than on
  2. BLAS backend — You may need OpenBLAS installed:
  # Using vcpkg
  vcpkg install openblas
  3. Limited testing — The project is primarily developed/tested on macOS. Some

  Quick Test

  # Query
  USE "hf://chrishayuk/gemma-3-4b-it-vindex";
  DESCRIBE "France";

  If you encounter issues, the CPU backend should work for all core functionality — just without GPU acceleration for inference.

  The Simple Explanation
                                                                                  LARQL lets you look inside an AI's brain and see what it "knows" — without
  asking it questions.

  ---
  The Problem It Solves

  Normally, to find out what an AI knows, you have to:
  1. Ask it a question
  2. Wait for it to think (run billions of calculations)
  3. Hope it gives you the right answer

  This is slow, expensive, and you can't be sure what it actually "knows" vs
  what it's making up.

  ---
  What LARQL Does Instead

  LARQL opens up the AI's brain (the model weights) and turns it into a
  searchable database.

  Think of it like this:

  ┌──────────────────────────────────┬──────────────────────────────────────┐
  │            Normal AI             │                LARQL                 │
  ├──────────────────────────────────┼──────────────────────────────────────┤
  │ "Hey AI, what's the capital of   │ Search database: "France" → finds    │
  │ France?"                         │ "capital: Paris"                     │
  ├──────────────────────────────────┼──────────────────────────────────────┤
  │ AI thinks for 500ms, uses GPU    │ Instant lookup, no GPU needed        │
  ├──────────────────────────────────┼──────────────────────────────────────┤
  │ You get an answer (maybe right,  │ You see exactly what's stored        │
  │ maybe wrong)                     │                                      │
  └──────────────────────────────────┴──────────────────────────────────────┘

  ---
  A Real Example

  -- Ask LARQL: "What does this AI know about France?"
  DESCRIBE "France";

  Output:
  France
    capital   → Paris     (stored in layer 27)
    language  → French    (stored in layer 24)
    continent → Europe    (stored in layer 25)

  You're literally seeing the "facts" encoded in the neural network's weights.

  ---
  Why Is This Useful?

  1. Debug AI Knowledge

  See exactly what a model knows (or doesn't know) about a topic.

  2. Edit AI Knowledge

  -- Add a new fact to the AI's brain
  INSERT INTO EDGES (entity, relation, target)
      VALUES ("Atlantis", "capital", "Poseidon");

  Now the AI "knows" that Atlantis's capital is Poseidon — without retraining.

  3. Fast Queries

  - Normal inference: 500ms, needs GPU
  - LARQL lookup: 0.3ms, no GPU

  4. Small Edits Instead of Retraining

  - Want to add 1000 company facts? Fine-tuning costs $$$
  - LARQL: Create a 10MB "patch" file, apply instantly

  AI models store knowledge as patterns in their weights. LARQL discovered that:

  1. Gate vectors = "What triggers this knowledge" (like a search index)
  2. Down vectors = "What this knowledge outputs" (like database values)

  By reorganizing these into a vindex (vector index), you can query them like a

> what is the magic that make LARQL work ?                                                                                                                      ● Great question. The "magic" is actually a deep insight about how transformers   store knowledge.

  ---                                                                             The Core Discovery

  Inside every transformer (GPT, Llama, Gemma, etc.), there are FFN layers
  (Feed-Forward Networks). Each layer does this calculation:

  output = down_projection(activation(gate(x) × up(x)))

  The breakthrough: Each row of the gate matrix paired with a column of the down
   matrix acts like a database row.

  ---
  Visualizing It

  Imagine the FFN layer has 10,000 "features". Each feature is:

  Feature #9515 (Layer 27):
  ┌─────────────────────────────────────────────────────┐
  │  GATE VECTOR (2560 numbers)                         │
  │  "Fires when input looks like: capital of France"   │
  ├─────────────────────────────────────────────────────┤
  │  DOWN VECTOR (2560 numbers)                         │
  │  "Outputs: push toward token 'Paris'"               │
  └─────────────────────────────────────────────────────┘

  The gate is the KEY. The down is the VALUE.

  It's literally a key-value store hiding inside matrix multiplication!

  ---
  The Magic Step-by-Step

  Step 1: The Model Learns Facts as Gate-Down Pairs

  During training, when the model sees "The capital of France is Paris"
  thousands of times, it learns:

  - Gate #9515: "Activate when I see 'capital of France' context"
  - Down #9515: "Add a vector that pushes logits toward 'Paris'"

  Step 2: LARQL Extracts These Pairs

  Instead of running the model, LARQL:
  1. Takes the gate matrix (what triggers each feature)
  2. Takes the down matrix (what each feature outputs)
  3. Reorganizes them into a searchable index

  Original Model:              LARQL Vindex:
  ┌─────────────────┐         ┌─────────────────┐
  │ gate_proj.weight│   →     │ KNN search index│
  │ (matrix multiply)│        │ (instant lookup)│
  └─────────────────┘         └─────────────────┘

  Step 3: Query = KNN Search

  When you ask DESCRIBE "France":

  1. Embed "France" → get a vector
  2. Find gate vectors most similar to that vector (KNN search)
  3. For each matching gate, look up what it outputs
  4. Return: "capital → Paris", "language → French", etc.

  No matrix multiplication. No forward pass. Just similarity search.

  ---
  Why This Works (The Math)

  In a normal forward pass:

  # Standard FFN computation
  gate_scores = input @ gate_matrix      # Matrix multiply (slow)
  activated = gelu(gate_scores * up_out)
  output = activated @ down_matrix       # Matrix multiply (slow)

  LARQL realizes:
  - input @ gate_matrix is just computing dot products between input and each
  gate row
  - High dot product = that feature is relevant
  - So skip the multiply — just find the top-K most similar gates

  # LARQL approach
  top_features = knn_search(input, gate_vectors, k=8000)  # Fast lookup
  output = sum(down_vectors[f] * score for f, score in top_features)

  Same result, different path. KNN replaces matmul.

  ---
  The Surprising Part

  This isn't an approximation. For factual queries, it gives identical results:

  INFER "The capital of France is" MODE walk   → Paris (97.91%)
  INFER "The capital of France is" MODE dense  → Paris (97.93%)

  Walk mode (LARQL's KNN approach) matches dense mode (standard matmul) because:
  - Only ~8,000 of 10,240 features actually fire significantly
  - The rest contribute near-zero
  - So finding the top-K is equivalent to computing all of them

  ---
  The Insight in One Sentence

  ▎ Transformers store knowledge as searchable key-value pairs hidden inside
  ▎ their weight matrices. LARQL just makes them searchable.

  ---
  Why Nobody Did This Before

  1. Scale: You need efficient KNN over millions of high-dimensional vectors
  2. Reorganization: The weights must be restructured (that's what a "vindex"
  is)
  3. Validation: Proving walk = dense required careful engineering

  LARQL cracked all three.

  Analogy

  ┌───────────────────────────────────────────────┬─────────────────────────┐
  │              Normal AI Inference              │          LARQL          │
  ├───────────────────────────────────────────────┼─────────────────────────┤
  │ Reading an entire encyclopedia to answer one  │ Using the index at the  │
  │ question                                      │ back                    │
  ├───────────────────────────────────────────────┼─────────────────────────┤
  │ O(all parameters)                             │ O(relevant features)    │
  ├───────────────────────────────────────────────┼─────────────────────────┤

  └───────────────────────────────────────────────┴─────────────────────────┘

  The "magic" is realizing the index was there all along — just encoded as
  matrix rows.
